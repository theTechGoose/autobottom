# Audit Execution

> What DID happen

---

## AuditBatch `stored`

Groups multiple AuditInstance entries into a single
batch submission. An admin or integration submits a
batch with field data for each record; the system
creates individual AuditInstance entries and tracks
batch-level progress.

| Field | Description |
| ----- | ----------- |
| `status` | BatchStatus (enum) |
| `instanceIds[]` | AuditInstance IDs in this batch |
| `doneIds[]` | completed instance IDs |
| `createdAt` | batch creation timestamp |
| `createdBy` | email of submitter |

**Relationships:**
- owns AuditInstance[] (instanceIds)

---

## BatchStatus `enum`

```
pending | running | paused | failed | retrying | finished
```

---

## AuditInstance `stored`

A single audit execution against one recording.

| Field | Description |
| ----- | ----------- |
| `status` | InstanceStatus (enum) |
| `subjectEmail` | person being audited |
| `configId` | `<uuid>:-:<version>` of audit config used |
| `batchId` | parent AuditBatch (null if standalone) |
| `recordingUrl` | link to original recording |
| `transcript` | TranscriptData (embedded) |
| `fieldValues` | `Record<string, unknown>` -- actual values keyed by FieldDef.key, passed at audit creation time |
| `results[]` | AuditResult (embedded) -- append-only, latest = current truth |
| `feedback` | FeedbackData (embedded, null until finalized) |

**Relationships:**
- references AuditConfig (configId)
- references AuditBatch (batchId, optional)

---

## AuditResult `embedded`

Appended to `AuditInstance.results[]`. Each entry represents a scoring pass
(LLM, reviewer, or judge).

| Field | Description |
| ----- | ----------- |
| `origin` | one of ResultOrigin names |
| `author` | `autobot+ai@monsterrg.com` for LLM, user email for human |
| `timestamp` | when produced |
| `answers[]` | AnswerEntry -- non-skipped questions |
| `skipped[]` | SkippedEntry -- skipped questions |

---

## ResultOrigin `enum`

```
llm | reviewer | judge
```

| Field | Description |
| ----- | ----------- |
| `questionId` | references AuditQuestion |
| `answer` | value conforming to AuditQuestion.questionType |
| `notes` | llm: thinking + defense, human: their reasoning |
| `ragDocs[]` | RagDoc -- retrieved documents (llm only) |
| `evaluation?` | AstResults -- scoring result (llm only) |

## SkippedEntry `embedded`

## InstanceStatus `enum`

```
pending | no-recording | creating-job | pulling-record |
getting-recording | transcribing | populating-questions |
asking-questions | resolved | appeal-pending | retrying
```

### Pre-transcription states

- `no-recording` -- recording could not be acquired
  from the recordings service (all retries
  exhausted)
- `creating-job` -- pipeline job being initialized
- `pulling-record` -- field data being fetched from
  data source
- `getting-recording` -- recording being downloaded
  from recordings service

---

## TranscriptData `embedded`

Three forms of the transcript are stored, each
produced at a different pipeline stage:

| Field | Description |
| ----- | ----------- |
| `raw` | plain text from transcription service |
| `diarized` | speaker-labeled transcript (speaker identification via LLM) |
| `words[]` | TranscriptWord (embedded) -- word-level with timestamps |

### Processing Pipeline

1. Transcription service produces raw text
2. LLM diarization labels speakers (agent vs
   customer vs other)
3. Word-level timestamps preserved for RAG
   retrieval and UI scrubbing

## Pipeline Mechanics

### Queue-Driven Steps

The pipeline is orchestrated via a queue service
(see services.md). Each step is an independent
endpoint that processes one stage and enqueues
the next:

```
init -> transcribe -> transcribe-complete ->
prepare -> ask-batch (fan-out) -> finalize ->
cleanup
```

### Fan-Out / Fan-In

The `ask-batch` step fans out questions into
parallel batches (configurable batch size). An
atomic counter tracks remaining batches. When the
last batch completes, the counter reaches zero and
`finalize` is enqueued.

### Error Handling & Retries

PipelineConfig controls retry behavior:

- `maxRetries` -- max retry attempts per step
  (default 5)
- `retryDelaySeconds` -- delay between retries
  (default 10)

Step-specific behavior:

- **Recording acquisition failure** -- status
  moves to `no-recording` after all retries
  exhausted. Audit is finalized as failed.
- **Transcription failure** -- re-enqueued with
  delay. After max retries, marked failed.
- **LLM transient errors** -- re-enqueued with
  exponential backoff for rate limits.
- **Storage failures** -- retried up to 5 times
  with delay.
- **Invalid recordings** -- all questions failed
  gracefully, audit finalized with error state.

All errors are tracked as pipeline statistics
(see logging spec).
